## Цель

M-04 положил `fts_log` (FTS5 keyword search). Vec embedding для layer4 пропустили — privacy + write-amp concerns. После M-04.1: rolling embedding окно последних N=10k log rows → episodic semantic search.

Privacy: log rows содержат raw user input до PII-scrub. **Vec embeddings stay agent-only** — не выкатывать на public REST. RAG layer "log" уже agent-only (M-04).

Write-amp: ~10k rows × NVIDIA embed → ~10k API calls. Fan-out batch'ами по 50, rate-limited через router. Один-раз initial backfill для existing logs + ongoing incremental на новые rows (через trigger или night-cycle step).

После M-04.1:
- Night-cycle step `embed-log.ts` — раз в сутки embed-ит новые log rows (rolling cap 10k самых свежих).
- `vec_embeddings` table получает rows с `layer='log'` (existing schema, just add usage).
- `RAGPipeline.search` для `layers: ["log"]` теперь использует BOTH FTS (M-04) AND vec (новое) → RRF merge → лучший recall.
- `bumpAccess` уже игнорирует "log" (M-02 isBumpLayer); leave as-is.

Foundation: improves LongMemEval multi-session recall — log retrieval по семантике, не только keyword.

## Изменение

### Rolling cap logic

Step запросит unembedded log rows up to N=10k самых свежих (`ORDER BY created_at DESC LIMIT 10000`), BATCH'ит embed-вызовы по 50 (NVIDIA limit-friendly), upsert'ит в `vec_embeddings(layer='log', id=log.id, embedding)`.

Eviction (rolling): когда vec_embeddings(layer='log') count > N=10k, DELETE самых старых (по created_at JOIN с layer4_log). Это поддерживает window.

```ts
const ROLLING_CAP = Number(process.env.LOG_EMBED_CAP) || 10000;
const BATCH_SIZE = 50;

async function runEmbedLog(deps): Promise<EmbedLogResult> {
  if (process.env.LOG_EMBED_ENABLED === "false") return zero;
  // 1. Find unembedded recent (≤ROLLING_CAP - existing).
  const rows = deps.memory.logRepo.selectUnembeddedRecent(ROLLING_CAP);
  let embedded = 0, errors = 0;
  // 2. Batch embed.
  for (const batch of chunks(rows, BATCH_SIZE)) {
    try {
      const vecs = await deps.rag.embedBatch(batch.map(r => r.content));
      deps.memory.db.transaction(() => {
        for (let i = 0; i < batch.length; i++) {
          deps.memory.upsertEmbedding(batch[i].id, "log", vecs[i]);
        }
      })();
      embedded += batch.length;
    } catch (err) {
      errors++;
      log.warn(`embed batch failed: ${(err as Error).message}`);
    }
  }
  // 3. Eviction — keep cap.
  const evicted = evictOldest(deps.memory, ROLLING_CAP);
  return { embedded, evicted, errors };
}
```

`upsertEmbedding(id, "log", vec)` — existing API (M-01 added shared layer; same signature works for "log").

### Eviction SQL

```sql
DELETE FROM vec_embeddings
 WHERE layer='log'
   AND id IN (
     SELECT v.id
       FROM vec_embeddings v
       JOIN layer4_log l ON l.id = v.id
      WHERE v.layer='log'
   ORDER BY l.created_at ASC
      LIMIT MAX(0, (SELECT count(*) FROM vec_embeddings WHERE layer='log') - ?)
   );
```

### RAG vec branch unblock

В `RAGPipeline.search` (M-04 fix-round dropped log из `vecLayers`):
```ts
// M-04: vecLayers = layers.filter(l => l !== "log");  // OLD
// M-04.1: vec branch teперь работает для log тоже (rolling embed).
const vecLayers = layers; // include "log" if requested
```

Но **только** если `layers` явно содержит "log" (default `["context","archive","shared"]` — не включает). Не break существующее поведение.

### `selectUnembeddedRecent` SQL

```sql
SELECT l.id, l.content, l.role
  FROM layer4_log l
 WHERE l.id NOT IN (SELECT id FROM vec_embeddings WHERE layer='log')
 ORDER BY l.created_at DESC
 LIMIT ?
```

`LIMIT` = `ROLLING_CAP - (current count layer='log')` чтобы не превышать window. Если result пустой → step no-op.

### Env knobs

- `LOG_EMBED_ENABLED` (default true). `false` → step no-op.
- `LOG_EMBED_CAP` (default 10000) — rolling window size.
- `LOG_EMBED_BATCH` (default 50) — batch size per embed call.
