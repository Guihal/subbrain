# Видение subbrain — drafted 2026-05-04

> Draft для дополнения. Структура — раздел = решение. Где `?` — открытый вопрос для юзера.

## TL;DR

Subbrain → personal **agent platform** уровня opencode/n8n hybrid. Бэк = LLM-gateway + agent-runtime + memory + MCP-registry. Фронт = единая консоль управления (роли, провайдеры, MCP, скиллы, задачи, ТГ-данные). Каждая капля — модуль с чёткой границей. Никакого монолита, никакого хардкода ролей.

## 1. Текущее vs целевое (one-screen)

| Слой | Сейчас | Куда |
|---|---|---|
| Gateway / proxy | `lib/model-router.ts` + per-role hardcode в `model-map.ts` | Standalone gateway: routing-config из БД, budgets, fallback policies, observability dashboard |
| Agent runtime | `pipeline/agent-loop` + `agent-pipeline` (pre/main/post) | Декомпозиция как opencode: primary/sub agents, mode-presets (Build/Plan/Research/Free), config-driven |
| Memory | 4 слоя (focus/shared/RAG/raw_log) + RAG hybrid + night cycle | Memory-v2 backlog (edges, salience, decay, dedup, kind) — поэтапно из `docs/tasks/memory-v2/` |
| MCP | Registry per-domain (`mcp/registry/*.tools.ts`) | + UI-панель: список серверов, health, scope-фильтры, hot-reload, install/uninstall из marketplace |
| Tasks | `agent_tasks` table (PR-C1..C4 backlog), free-runner pool | UI-таблица с filters/Kanban, runner-types выбираются из конфига, history+digest |
| Frontend | Nuxt 4, 4 pages (chat / memory / tasks / freelance), под рефакторингом | Полный rewrite: workspace-shell + панели (роли / провайдеры / MCP / скиллы / задачи / ТГ-data / memory) |
| Telegram | userbot + bot, polling (DONE), digest (PR-C3) | Расширить как **data-source**: read-only ingest рабочих чатов → структурированный store + поиск |
| Workflow automation | — | n8n как side-car: рутины (cron, webhooks, ETL) вне agent-loop'а |

## 2. Архитектурное деление (opencode-style + LLM-gateway add-on)

Идея — **client/server split**, как у opencode (см. [Intro | OpenCode](https://opencode.ai/docs/)). Бэк-сервер = единственная точка истины, фронт = один из клиентов.

```
┌────────────────────────────────────────────────────┐
│ Clients                                            │
│  ├─ Web UI (Nuxt rewrite)                          │
│  ├─ Telegram bot/userbot                           │
│  ├─ MCP-servers (subbrain-as-mcp для CC/Codex)     │
│  └─ Future: VSCode ext, mobile, CLI                │
└──────────────────┬─────────────────────────────────┘
                   │ HTTP + SSE (Elysia)
┌──────────────────▼─────────────────────────────────┐
│ View / Transport (routes/, mcp/transport.ts)       │
│  TypeBox validation → delegate to logic            │
└──────────────────┬─────────────────────────────────┘
┌──────────────────▼─────────────────────────────────┐
│ Logic (services/, pipeline/, mcp/tools/, scheduler)│
│  AgentService, ChatService, MemoryService,         │
│  AgentPool, AgentLoop, AgentPipeline               │
└────────┬──────────────────────┬────────────────────┘
         │                      │
┌────────▼──────────┐ ┌─────────▼───────────────────┐
│ Data (repos/, db/)│ │ Outbound: LLM Gateway       │
│ SQLite + FTS+vec  │ │  (model-router → provider)  │
└───────────────────┘ └─────────┬───────────────────┘
                                │
                                ▼
                        Providers: NIM / OpenRouter
                        / Copilot / cliproxy / future
```

3-layer SoC уже зашит в `subbrain-guardrails` skill (data/logic/view, file-cap 150). Дальнейшие фичи строго по этой схеме.

## 3. LLM Gateway (агент-прокси) — отдельный модуль

Сейчас `ModelRouter` смешан с pipeline. Цель — **standalone gateway** уровня litellm/portkey, но встроенный (single-Bun-process), не sidecar. Важно: для **личного** use-case'а внешний gateway-сервис избыточен.

### Что вынести в `src/gateway/`:

- **Routing config** — таблица `provider_routes` в БД (читается вместо `lib/model-map.ts` const'ы). Поля: `virtual_role`, `provider`, `model_id`, `priority`, `fallback_chain` (JSON), `budget_limit_daily`, `disabled`.
- **Per-key budgets** (как у LiteLLM, см. [LiteLLM AI Gateway docs](https://docs.litellm.ai/docs/simple_proxy)) — отслеживать $/токены per `chat_id` / `task_id` / `runner_type`. Лимиты → 429 с явным `quota_exhausted` error.
- **Cost tracking** — `request_log` table расширить полями `prompt_tokens`, `completion_tokens`, `cost_usd`, `provider`, `latency_ms`. Aggregated dashboard в UI.
- **Observability** — built-in tracing per request (parent → child agent calls visible). Не Langfuse/Helicone, а простой viewer в UI.
- **Failover policy** — конфиг per role. Сейчас hardcoded в `model-router`. Выносим в БД → UI меняет без redeploy.
- **Caching** (опционально, low-prio) — semantic cache для repeated questions. Реализуется через RAG-lookup на pre-step.

### Что НЕ вынести (намеренно):

- Provider implementations (NIM, OpenRouter, Copilot, cliproxy) — остаются в `providers/`. Gateway = orchestration layer над ними.
- Rate-limiter — уже atomic под Mutex (см. guardrail #4). Расширить per-key lookup, не переписывать.

### UI-панель «Providers»

```
┌─────────────────────────────────────────────────────┐
│ Providers                                           │
│ ┌─NVIDIA NIM─────┐ ┌─OpenRouter────┐ ┌─Copilot─────┐│
│ │ ✓ healthy      │ │ ✓ healthy     │ │ ✗ stale     ││
│ │ 12/40 RPM      │ │ 0 spent today │ │ token expired││
│ │ $0.00 today    │ │ $1.24 today   │ │ [reauth]    ││
│ │ [models] [edit]│ │ [models][edit]│ │             ││
│ └────────────────┘ └───────────────┘ └─────────────┘│
│ + Add provider                                      │
└─────────────────────────────────────────────────────┘
```

Add provider — wizard: тип (OpenAI-compat / Anthropic / NIM / custom), URL, ключ, пробный ping, sync моделей в `provider_routes`.

## 4. Frontend full rewrite

Текущий: Nuxt 4, 4 страницы, рефакторинг по wave-плану в [refactor/28-file-size-150-limit.md](tasks/refactor/28-file-size-150-limit.md). Юзер: «ебаный сейчас» → rewrite целиком, на той же платформе (Nuxt 4 ok), но с переработанной UX-моделью.

### Принципы

1. **Workspace shell** — вместо отдельных pages навигация через persistent sidebar + tabbed workspace (как в opencode TUI или Linear).
2. **Каждая панель = self-contained module** под `web/app/modules/<name>/`. Свой routing, store, components, composables. Lazy-loaded.
3. **Без glue-кода в pages** — pages = только mount + mode switching.
4. **Realtime first** — все mutations через SSE/WS broadcast → панели обновляются автоматически. Не polling.

|---|---|---|

| `roles` | Создание / редактирование virtual roles. См. §4.1 | New |
| `providers` | См. §3 UI-панель | New |

### 4.1 Roles module (UI создания ролей)

Сейчас роли = hardcode в `lib/model-map.ts`. Цель — БД-driven с UI-редактором.

```
┌─ Roles ──────────────────────────────────────────────┐
│ teamlead    glm-5.1 (NIM)        →fallback:MiniMax  │
│ coder       deepseek-v4 (NIM)    →fallback:qwen3   │
│ critic      glm-5.1                                 │
│ memory      deepseek-v4-flash    + skill:hippocampus│
│ chaos       kimi-k2.6                               │
│ + Create role                                       │
└─────────────────────────────────────────────────────┘
```

**Create role wizard:**
- Name (slug, валидация unique)
- System-prompt (markdown editor + variables: `{{date}}`, `{{focus}}`, `{{shared}}`)
- Provider/model (dropdown из `provider_routes`)
- Fallback chain (drag-reorder)
- Tool scope (multi-select из MCP registry — см. §4.2)
- Skills attached (multi-select из §4.3)
- Mode (`primary` / `subagent` — для agent-teams)
- Permissions per tool: `allow` / `ask` / `deny` (как у opencode, см. [Agents | OpenCode](https://opencode.ai/docs/agents/))
- Temperature, max_steps, idle_timeout

DB schema: `roles` table (replaces map), `role_tool_scopes`, `role_skills`. Migration: seed существующие 7 ролей при boot если table пуста.

### 4.2 MCP servers panel

Сейчас MCP registry — статика в `mcp/registry/*.tools.ts`. Цель — динамическая registration + scope-management через UI.

```
┌─ MCP Servers ────────────────────────────────────────┐
│ ✓ playwright       built-in    23 tools  [scope...]│
│ ✓ telegram         built-in    8 tools   [scope...]│
│ ✓ memory           built-in    5 tools   [scope...]│
│ ✓ subbrain-self    built-in    12 tools  [scope...]│
│ ○ github           external    [install] from npm  │
│ ○ linear           marketplace [install]           │
│ + Add MCP server (URL / npm / docker)              │
└─────────────────────────────────────────────────────┘
```

- **Scope editor** per server: per role + per agentMode (`scheduled` / `interactive` / `pool-runner`). Scoped tools отрезаются registry filter'ом, не runtime if (см. guardrail #15).
- **Tool inspector** — список tools, schema, last-call latency, error rate, scope usage.
- **Install flow** — для npm-MCP: `npx @org/server` запуск как child process под supervision. Для remote — config endpoint URL + auth.
- **Health** — heartbeat ping, restart on failure.

### 4.3 Skills для модели

Идея аналогична Claude Code skills (`.claude/skills/<name>/SKILL.md`) — pre-formatted prompt fragments, attachable к ролям/runs.

```
┌─ Skills ─────────────────────────────────────────────┐
│ subbrain-guardrails    md   attached to: 4 roles    │
│ telegram-handler       md   attached to: 1 role     │
│ rag-search-strategy    md                           │
│ sql-migration-rules    md                           │
│ + New skill                                         │
└─────────────────────────────────────────────────────┘
```

DB: `skills` table — `name`, `description`, `body` (md), `auto_invoke_keywords` (массив для auto-trigger), `tools_required`. Attachment через `role_skills` или `chat_skills` (per-conversation override).

**Attach mechanism:** при инжекте system-prompt'а — concat'нуть body'ы attached skills + role.system_prompt + memory layers. Single source of truth — `services/prompt-assembler.ts`.

### 4.4 TG data ingest

Юзер: «сбор данных с рабочих тг аккаунтов». Цель — userbot читает рабочие чаты в read-only, складывает в structured store, фронт даёт search/filter/digest.

**Уже есть:** `src/telegram/userbot.ts` (MTProto), `tg_read_chat` MCP tool, polling из задачи 02.

**Чего не хватает:**
- **Persistent ingest** — поток сообщений в `tg_messages` table (chat_id, msg_id, from, ts, text, reply_to, media_refs). Index на (chat_id, ts).
- **Per-chat policy** — UI с чекбоксами: «ingest on/off», «redact PII», «extract tasks (LLM)», «notify on keyword». Хранится в `tg_chat_policy`.
- **Search UI** — full-text по `tg_messages` (FTS5 уже есть в проекте). Filter: chat / sender / date-range / has-task.
- **Digest mode** (`digest.ts` из PR-C3) расширить — daily rollup per workspace, не только pool-tasks.
- **Task extraction** — pipeline-job: LLM пройдётся по новым сообщениям, выделит actionable items → `agent_tasks` (work scope) → users review в UI.

**Privacy / safety:** на старте — opt-in per chat, default off. Data только локально в SQLite. ⚠️ Юзер: рабочие аккаунты — есть риск NDA-чувствительных данных. До deploy на prod надо отдельный consent-flow.

### 4.5 Tasks table (расширение)

Текущий `tasks.vue` + `agent_tasks` (PR-C1..C4). Add:
- **Project field** — задача → проект (новая table `projects`). Filter / Kanban per project.
- **Source** — manual / pool-generated / extracted-from-tg / extracted-from-chat.
- **Assignee** — user или agent-role.
- **Artifact viewer** — `done_with_artifact` уже даёт payload; UI рендерит markdown / diff / link.
- **Cron tasks** — recurring schedule (e.g. «каждый понедельник 10:00 — research + digest»). Хранится в `task_schedules`, executor — n8n (см. §6) или встроенный scheduler.

## 6. n8n integration (research)

См. [n8n.io](https://n8n.io/), [n8n AI Agents](https://n8n.io/ai-agents/), [n8n self-hosted Docker guide](https://dev.to/jangwook_kim_e31e7291ad98/how-to-self-host-n8n-with-docker-ai-workflow-automation-guide-2026-3lec).

### Что есть у n8n

- 400+ integrations (Slack, Notion, GH, Postgres, S3, Linear, Discord, Gmail, Calendar, etc.).
- AI Agent builder: LLM chains, vector store nodes (Qdrant/Pinecone), tool-using agents с memory+guardrails.
- Self-hosted Docker, cron triggers, webhooks, RBAC, audit logs.
- 220 exec/sec single instance.

### Где n8n нужен subbrain'у

**ДА** (n8n side-car через docker-compose):
- **Cron-задачи рутины** — например «каждое утро 8:00 → fetch GH issues → summary в TG». Сейчас такое лезет в `agent-pool` с runner'ом, что overkill. n8n даёт визуальный flow + 400+ integrations без кодинга.
- **Webhook-receivers** — внешние сервисы шлют события в n8n → n8n → subbrain `/v1/chat` или `agent_tasks_enqueue`. Subbrain не знает про каждый внешний сервис, n8n знает.
- **ETL pipelines** — periodic data ingest (e.g. Linear issues → memory), без необходимости писать MCP-tool.
- **Notification fan-out** — subbrain emit'ит event → n8n шлёт в TG / Slack / email с per-channel formatting.

**НЕТ** (subbrain делает лучше):
- Agent-loop / reasoning / tool-calling — это core subbrain.
- Memory layers — n8n vector-stores не replacement для 4-layer memory + RAG hybrid.
- MCP integration — у subbrain уже свой registry с scope filtering.
- Anything с custom tool-writing (`create_code_tool`) — n8n nodes не дают такой гибкости.

### Предложение интеграции

```
docker-compose:
  subbrain:    # main (Bun:4000)
  n8n:         # workflow runner (web:5678)
  # subbrain ←HTTP→ n8n через webhook nodes в обе стороны
```

- В subbrain UI: модуль **Workflows** = embedded iframe с n8n editor + список «активных flows связанных с subbrain». Не дублируем n8n UI, just deep-link.
- На subbrain side — два MCP tools: `n8n_trigger(workflow_id, payload)` и `n8n_list_workflows()`. Агент может запустить готовый flow.
- На n8n side — node «subbrain» (custom): шлёт promptна `/v1/chat`, читает из `/v1/memory/search`, пишет в `agent_tasks`.

**Альтернатива (если n8n overkill):** Temporal / Inngest / простой in-house cron-engine. Решение оставить открытым (?) — но n8n даёт integrations bonus.

## 8. Open вопросы для дополнения юзером

- **Multi-user / single-user?** Сейчас всё single-user (один токен в `auth.ts`). Если multi — нужна workspaces table, RBAC, isolation. Влияет на schema всех модулей.
- **Mobile-first или desktop-first** для rewrite фронта? Сейчас web-only desktop. Telegram closes часть mobile-кейсов.
- **n8n side-car или встроенный workflow engine?** n8n легко поднять, но docker overhead + чужой UI. Альтернатива — встроенный визуальный flow editor (huge work).
- **Where to host?** Сейчас VPS 109.120.187.244 + Caddy. С rewrite фронта — оставляем там же или Vercel/Cloudflare для UI + VPS для бэка?
- **Public / private?** Subbrain доступен только тебе или планируется invite-only для друзей / public с paywall?
- **Codex / Claude Code integration уровень?** subbrain-as-MCP уже работает (CC ↔ subbrain). Расширять (e.g. CC видит agent-tasks, ставит свои)?
- **Voice?** TG voice → STT → agent → TTS. Сейчас вне scope. Включать в roadmap?
- **Cost ceiling?** Какой бюджет/мес на API. Влияет на default fallback policy (более агрессивный fallback на free-tier при low budget).
- **Каких MCP-серверов из marketplace тебе явно не хватает прямо сейчас?** (GitHub, Linear, Notion, Gmail/Calendar — что в первую очередь.)

## 9. Что НЕ делаем (anti-scope)

- **Не пишем свой LLM** — провайдеры остаются external.
- **Не делаем training/fine-tuning UI** — задача отдельного проекта.
- **Не делаем замену n8n** — если выбрали n8n, embed его, не клонируем.
- **Не строим свой vector DB** — sqlite-vec достаточно для personal scale.
- **Не делаем k8s** — single-VPS Docker-compose до явного scaling-event.
- **Не пилим plugin system на JS-eval** — security-кошмар; вместо — strict schema-driven config + MCP-extensions.
